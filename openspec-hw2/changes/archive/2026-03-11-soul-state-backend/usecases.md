## 用例

### 用例：获取灵魂状态

**主要参与者：** 客户端（Web / Android / 鸿蒙）
**范围：** ergouWeb 后端
**层级：** 用户目标

**利益相关者：**
- 用户 — 在所有设备上看到一致的二狗性格
- 客户端 — 获取灵魂状态用于展示和本地缓存

**前置条件：**
- 用户已认证（有效 Session）

**成功保证：**
- 客户端收到完整灵魂状态：7 个参数 + 关系阶段 + 交互计数 + 更新时间

**触发条件：** 客户端发送 `GET /api/soul-state`

**主成功场景：**
1. 客户端请求当前用户的灵魂状态。
2. 系统查找该用户的灵魂状态记录。
3. 系统返回完整灵魂状态及 `updated_at` 时间戳。

**扩展：**
- 2a. 该用户无灵魂状态记录：
  - 2a1. 系统以默认值创建一条灵魂状态记录（stranger 阶段）。
  - 2a2. 系统返回新创建的默认状态。

### 用例：手动调整灵魂状态

**主要参与者：** 用户（通过客户端或管理后台）
**范围：** ergouWeb 后端
**层级：** 用户目标

**利益相关者：**
- 用户 — 希望重置或自定义性格参数
- 管理员 — 需要为用户重置灵魂状态（客服场景）

**前置条件：**
- 用户已认证

**成功保证：**
- 指定字段已更新；未指定字段保持不变
- 演化日志记录每个变更的参数，trigger_type 为 "manual"
- 返回更新后的完整状态

**触发条件：** 客户端发送 `PUT /api/soul-state`，携带部分字段

**主成功场景：**
1. 客户端提交一个或多个待更新的灵魂状态字段。
2. 系统校验每个字段值是否在合法范围内。
3. 系统更新指定字段，其余字段不变。
4. 系统将每个变更的参数写入演化日志。
5. 系统返回更新后的完整灵魂状态。

**扩展：**
- 2a. 字段值超出范围（如 `warmth_level: 1.5`）：
  - 2a1. 系统返回 400，列出无效字段及合法范围。
- 2b. 包含未知字段名：
  - 2b1. 系统忽略未知字段，仅处理有效字段。

### 用例：对话后演化灵魂状态

**主要参与者：** 系统（对话完成后内部触发）
**范围：** ergouWeb 后端
**层级：** 子功能

**利益相关者：**
- 用户 — 性格随交互逐渐适应
- 系统 — 维护演化护栏（频率限制、变化上限）

**前置条件：**
- `POST /api/chat` 刚完成一轮对话
- 用户已有灵魂状态记录

**成功保证：**
- `total_interactions` 加 1
- 若触发演化：参数调整在 ±0.05 以内，记录到演化日志
- 若交互数越过关系阈值：`relationship_stage` 升级
- 当日演化次数不超过 10 次

**触发条件：** `POST /api/chat` 成功返回响应后

**主成功场景：**
1. 对话处理器通知一轮对话已完成。
2. 系统将 `total_interactions` 加 1。
3. 系统检查该用户当日演化次数。
4. 系统通过 LLM 分析对话内容，确定参数调整方向。
5. 系统应用调整（每个参数 ±0.05 上限，钳制在字段合法范围内）。
6. 系统检查 `total_interactions` 是否越过关系阈值。
7. 系统将所有变更写入演化日志。
8. 系统持久化更新后的灵魂状态。

**扩展：**
- 3a. 当日演化已达 10 次上限：
  - 3a1. 仍递增 `total_interactions` 并检查关系阈值。
  - 3a2. 跳过 LLM 分析和参数调整。
- 4a. LLM 分析失败或超时：
  - 4a1. 系统记录错误但不阻塞，灵魂状态保持不变。
  - 4a2. `total_interactions` 仍然递增。
- 6a. 越过关系阈值（如 10 次 → 相识）：
  - 6a1. 系统升级 `relationship_stage` 并记录此转变。

### 用例：基于灵魂状态构建系统 Prompt

**主要参与者：** 系统（对话处理内部流程）
**范围：** ergouWeb 后端
**层级：** 子功能

**利益相关者：**
- 用户 — 体验到与交互历史一致的性格
- LLM — 收到结构完整、反映当前灵魂状态的系统 Prompt

**前置条件：**
- 用户已认证并发起对话
- 灵魂状态记录存在（或首次 GET 时懒创建）

**成功保证：**
- 组装出完整的系统 Prompt，包含 `llm/system-prompt.md` 的静态部分 + 灵魂状态派生的动态部分

**触发条件：** `POST /api/chat` 开始处理用户消息

**主成功场景：**
1. 系统从数据库读取用户的灵魂状态。
2. 系统加载静态人设模板（身份、基础性格、引经据典、说话方式、记忆指令、安全边界）。
3. 系统根据 `warmth_level`、`trust_level`、`classical_ratio`、`verbosity_level`、`proactivity_level` 生成动态性格段落。
4. 系统根据 `classical_ratio` 生成动态说话风格段落（文白比例）。
5. 系统根据 `relationship_stage` 生成动态行为段落。
6. 系统生成与当前关系阶段匹配的语气示例。
7. 系统将静态 + 动态部分拼装为最终 Prompt。
8. 系统将拼装好的 Prompt 作为 system message 传给 LLM。

**扩展：**
- 1a. 无灵魂状态记录：
  - 1a1. 系统创建默认灵魂状态（同"获取灵魂状态"用例扩展 2a）。

### 用例：注册时初始化灵魂状态

**主要参与者：** 系统（注册流程触发）
**范围：** ergouWeb 后端
**层级：** 子功能

**利益相关者：**
- 新用户 — 获得一致的初始体验（stranger 阶段、默认性格）

**前置条件：**
- 用户正在通过 `POST /api/auth/register` 注册

**成功保证：**
- 新用户拥有一条默认值的灵魂状态记录

**触发条件：** 用户注册成功

**主成功场景：**
1. 注册处理器创建用户账户。
2. 系统为新用户插入一条默认灵魂状态记录。
3. 注册正常完成。

**扩展：**
- 2a. 灵魂状态插入失败（如约束冲突）：
  - 2a1. 系统记录错误。注册仍成功——灵魂状态将在首次 GET 时懒创建。
